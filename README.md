# 长寿经济产业图谱

一个面向长寿经济产业链上下游企业的开源产业图谱原型，并扩展了地域产业聚集地案例、全球长寿科技企业与中国银发健康上市公司的观察样本。目标是把上游研发、中游产品、下游服务、支撑生态、地域产业园区和可持续研究监测串成可查询、可扩展的产业链路。

这个项目同时追求两种价值：内容价值是让企业和研究者更快找到产业链资源；情绪价值是让关心健康寿命的人看到自己可以参与的技术、产品、城市和照护议题。

## 为什么做

长寿经济不是单一养老服务，也不是单一抗衰技术。更有价值的视角，是把健康寿命延长拆成研发源头、药械与数字产品、医疗康养服务、支付保险、政策资本和城市/园区场景。当前主图谱明确拆分为上游、中游、下游、支撑四类，每类 4 个节点，共 16 个节点。

苏州生物医药产业园 BioBAY 已作为第一个地域产业聚集地标杆案例加入网站。它不是替代主图谱，而是展示一个生物医药园区如何把创新药、高端医疗器械、IVD、生物技术、AI+BT、生物制造和康养机构场景，转化为长寿经济的区域样板。

## 当前功能

- 产业链图谱：按上游、中游、下游、支撑筛选，并可点击 16 个节点查看对应能力与资源。
- 产业聚集地案例库：将苏州 BioBAY 作为首个标杆案例，并预留上海张江、北京中关村/昌平生命谷、深圳/粤港澳、海南博鳌乐城、成都/成渝生命健康带等入口。
- 合作需求雷达：把创新药、IVD、器械、AI+BT、康养机构、城市园区的常见问题转成可查找的资源和补证据任务。
- 观察清单：展示全球长寿科技企业 3 家与中国市场上市企业 3 家，作为研究原型中的关注样本，并保留来源链接。
- 日报模板示例：沉淀公开信息监测结构，覆盖新闻、公告、融资、临床、合作、股价和风险变化；不代表当前已经每日运营。
- 初始研报：[全球长寿经济市场分析与三家高成长企业](./全球长寿经济市场分析与三家高成长企业.md)。
- 企业目录：展示产业链能力类型、标签和长寿经济合作需求。
- 资源目录：整理园区行动计划、官方介绍、AI4Science、大健康政策等入口。
- 证据模型：把公开事实、产业连接判断和待验证问题分层，便于审阅与继续补充。
- 静态部署：不依赖后端，可直接用于 GitHub Pages。

## 运行方式

直接在浏览器打开 `index.html`。

也可以在本地启动一个静态服务：

```bash
python3 -m http.server 8080
```

然后访问 `http://localhost:8080`。

## 本地自检

提交前建议运行。若本地有 npm：

```bash
npm run validate
```

没有 npm 时可以直接运行 `node scripts/validate-static-site.mjs`。它会检查核心文件、BioBAY 标杆案例、合作需求、证据模型、贡献模板、GitHub issue 模板、GitHub Actions 配置和 `app.js` 语法。仓库已配置 GitHub Actions，会在 push 和 pull request 时自动运行同一套校验。

## GitHub Pages 发布

仓库已包含 `.github/workflows/pages.yml`。推送到 `main` 后，GitHub Actions 会先运行 `npm run validate`，再通过 GitHub Pages 部署当前静态站点。

如果是第一次启用 Pages，请在 GitHub 仓库的 Settings → Pages 中选择 GitHub Actions 作为发布来源。

## 建议的数据结构

企业或机构记录建议包含：

- `name`：企业或机构名称
- `node`：所属产业节点
- `stage`：上游、中游、下游或支撑
- `location`：地区或能力类型
- `need`：希望寻找的合作资源
- `tags`：能力标签
- `source`：公开来源链接
- `updatedAt`：更新时间
- `contributor`：贡献人或组织

资源记录建议包含：

- `kind`：政策、报告、数据库、标准、城市试点、融资事件等
- `title`：资源标题
- `summary`：一句话价值说明
- `node`：关联产业节点
- `url`：来源链接
- `updatedAt`：更新时间

合作需求记录建议包含：

- `actor`：提出需求的企业、机构或城市角色
- `node`：关联产业节点
- `stage`：上游、中游、下游或支撑
- `question`：这个角色最需要回答的产业问题
- `seek`：需要寻找的场景、政策、支付、临床、资本或技术伙伴
- `resource`：当前图谱中可以先查看的资源
- `nextStep`：下一步需要补充的证据

产业聚集地记录建议包含：

- `name`：园区、城市或产业带名称
- `region`：所在区域
- `status`：标杆案例、观察中、征集中等
- `role`：在长寿经济中的产业角色
- `assets`：已有产业资产
- `gaps`：下一步验证方向
- `links`：案例、政策、报告或贡献入口

标杆案例建议继续补充：

- `metrics`：企业规模、重点集群、政策工具、转化方向等摘要指标
- `representatives`：代表企业或主体，必须能链接到公开来源
- `scenarioChains`：早筛、治疗、康复、产业化等转化链
- `policyTools`：行动计划、目录、医保商保、审评审批、资本、国际化工具
- `caseRows`：企业/主体、所属节点、产品或能力、长寿经济价值、当前证据、来源链接、待验证问题
- `applicationScenarios`：应用场景、连接节点、可导入产品、可衡量结果、需要补充的数据

## 证据模型

为了让图谱既有观点也能被审阅，建议把每条新增内容拆成三层：

- 事实层：官方介绍、政策文件、企业公告、上市产品、融资、临床进展等，用来证明主体、产品、政策或交易确实存在。
- 连接层：说明这个主体如何连接早筛、治疗、康复、照护、支付、出海或产业化链路，用来解释它为什么属于长寿经济。
- 验证层：保留待验证问题、来源链接、更新时间、贡献者和下一步要补的证据，避免把推测写成事实。

贡献内容时，优先写清楚“当前证据是什么”和“还有什么没有验证”。这样后续企业、园区、研究者和开发者可以继续接力，而不是只留下一个无法追溯的名单。

## 数据状态与更新节奏

- 已验证：BioBAY 标杆案例、16 个产业节点、18 个资源入口优先采用园区官网、政府文件、企业公告和公开报道。
- 待验证：其他城市园区、康养采购清单、支付转化效果、产品真实世界数据，需要继续补公开来源和案例。
- 建议频率：政策和园区动态按月复核，企业公告、临床、融资、出海 BD 和采购试点按周或事件驱动更新。

- 产品案例：真实产品或机制的名称、适应症或场景、当前证据、注册/上市/采购/医保商保状态，以及它对健康寿命延长的价值。

请把不确定的内容明确写成待验证问题。这个项目宁可先留下一个清楚的问题，也不要把推测包装成事实。

- 网站代码使用 MIT License，见 `LICENSE`。
- 产业数据建议使用 CC BY 4.0，并要求贡献者提供来源链接。
